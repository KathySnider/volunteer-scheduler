## Overview

This document describes the implementation of passwordless email authentication (magic links) in the `volunteer-scheduler` project. The implementation was ported from the `aarp-volunteer-system` (Next.js/NextAuth) and adapted for volunteer-scheduler's Go/GraphQL backend and Next.js 16/JavaScript frontend.

**Key Features:**
- One-time magic link tokens sent via email
- Automatic transport selection: Resend API (production/real emails) or Mailhog SMTP (development)
- Session management via database-stored tokens
- Rate limiting (5 requests per hour per email)
- 15-minute token expiry
- Frontend sign-in flow with cross-tab session detection
- React auth context with localStorage-backed sessions

## Architecture

### Authentication Flow

```
1. User visits /auth/signin
2. User enters email, clicks "Sign in with Email"
3. Frontend calls requestMagicLink GraphQL mutation
4. Backend generates token, stores in magic_links table, sends email
5. User clicks magic link in email → opens /auth/magic-link?token=<TOKEN>
6. Frontend calls consumeMagicLink GraphQL mutation
7. Backend validates token, creates session, returns sessionToken
8. Frontend stores session in localStorage via AuthContext
9. Sign-in tab detects localStorage change, shows "You can close this tab"
10. User clicks "Go to Events" on magic link tab → redirects to /
```

### Components

#### Database Schema (`database/migrations/`)
- `000002_magic_links.up.sql` — Stores magic link tokens
  - Fields: `id`, `email`, `token`, `created_at`, `expires_at`, `used_at`, `ip_address`, `user_agent`
  - Indexes on token and email for performance

- `000003_sessions.up.sql` — Manages user sessions
  - Fields: `id`, `email`, `token`, `created_at`, `expires_at`, `last_activity_at`

#### Backend Services (`backend/services/`)

**`mailer.go`** — Email transport abstraction
- `Mailer` struct wraps transport selection
- `ResendTransport` — Uses Resend API (`https://api.resend.com/emails`)
- `MailhogTransport` — Uses local SMTP (configurable host:port)
- Transport selection logic:
  - `USE_RESEND=true` → Resend
  - `NODE_ENV=production` → Resend
  - Otherwise → Mailhog

**`auth_magiclink.go`** — Token lifecycle management
- `MagicLinkService` — Main service with methods:
  - `GenerateMagicLink(ctx, email, ip, userAgent)` — Create & store token
  - `SendMagicLinkEmail(ctx, to, token)` — Send branded HTML email with inline styles
  - `ConsumeMagicLink(ctx, token)` — Validate & mark token as used
  - `CreateSessionToken(email)` — Generate session token, store in sessions table
  - `ValidateSessionToken(ctx, token)` — Check session validity
  - `CleanupExpiredTokens(ctx)` — Periodic cleanup job

#### GraphQL Schema (`backend/graph/volunteer/schema.graphql`)

```graphql
type Mutation {
  requestMagicLink(email: String!): MagicLinkResult!
  consumeMagicLink(token: String!): AuthResult!
}

type MagicLinkResult {
  success: Boolean!
  message: String!
  email: String
}

type AuthResult {
  success: Boolean!
  message: String!
  email: String
  sessionToken: String
}
```

#### Frontend Auth (`frontend/src/app/`)

**`context/AuthContext.js`** — Session state management
- Stores `{ email, sessionToken }` in localStorage under `authSession` key
- Provides `useAuth()` hook for all components
- `signOut()` clears both `authSession` and legacy `currentVolunteer`

**`auth/signin/page.js`** — Sign-in page
- Email input form → calls `requestMagicLink` mutation
- "Check your email!" confirmation after success
- Listens for `storage` event to detect when magic link is consumed in another tab
- Transitions to "You're signed in! You can close this tab." when detected

**`auth/magic-link/page.js`** — Magic link callback
- Reads token from URL, calls `consumeMagicLink` mutation
- Shows confirmation with "Go to Events" button (waits for user click)
- Error state with "Try Again" link

**`components/VolunteerEventsApp.jsx`** — Main app integration
- Redirects to `/auth/signin` if not authenticated
- Shows user email and Sign Out in sidebar when authenticated
- Existing volunteer name modal preserved for shift signup identity

## Environment Configuration

### `.env` Variables

```bash
# Database
DATABASE_URL="postgresql://user:password@127.0.0.1:5433/volunteer-scheduler"

# Email — Magic Link
RESEND_API_KEY=re_your_api_key_here
EMAIL_FROM=noreply@your-verified-domain.com
USE_RESEND="false"              # "true" = Resend, "false" = Mailhog

# Mailhog (used when USE_RESEND="false")
EMAIL_SERVER_HOST="localhost"
EMAIL_SERVER_PORT="1025"
```

### Docker Compose

The API service reads email config from the `.env` file via variable substitution:
```yaml
environment:
  EMAIL_FROM: ${EMAIL_FROM:-noreply@volunteer-scheduler.local}
  USE_RESEND: ${USE_RESEND:-false}
  RESEND_API_KEY: ${RESEND_API_KEY:-}
  EMAIL_SERVER_HOST: ${EMAIL_SERVER_HOST:-host.docker.internal}
  EMAIL_SERVER_PORT: ${EMAIL_SERVER_PORT:-1025}
```

**Important:** Changing email settings in `.env` requires restarting the API container (see Developer Guide below).

## Security Considerations

### Token Security
- **Generation:** 32 random bytes (64 hex chars) via `crypto/rand`
- **Storage:** Indexed in PostgreSQL for fast lookups
- **Expiry:** 15 minutes
- **Single-use:** Marked as `used_at` after consumption

### Rate Limiting
- 5 magic link requests per email per hour
- Checked at token generation time

### Session Management
- Session tokens are random 32-byte strings stored in the `sessions` table
- Default expiry: 30 days (configurable via `SESSION_MAX_AGE`)
- Currently stored in localStorage on the frontend (future: HttpOnly cookies)

### Known Limitations
- Session tokens are stored in localStorage (not HttpOnly cookies) — acceptable for dev, should be upgraded for production
- No server-side session validation on GraphQL queries yet — the session token is stored but not sent as an Authorization header
- The volunteer name modal and email-based auth coexist as separate identity systems — a future enhancement should link email to volunteer records

## Future Enhancements

1. **Link email to volunteer record** — Eliminate the name modal by matching authenticated email to a volunteer in the database
2. **Server-side session validation** — Send session token as Authorization header with GraphQL requests; validate in middleware
3. **HttpOnly cookies** — Move session token from localStorage to secure cookies
4. **Protected routes** — Add route guards that require authentication
5. **Account request/approval workflow** — Port from aarp-volunteer-system
6. **Audit logging** — Track login attempts, failed auths, IP patterns
7. **Token cleanup job** — Schedule periodic `CleanupExpiredTokens` (daily cron)
